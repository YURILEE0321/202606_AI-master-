### Agent 페르소나 및 시스템 프롬프트 (Identity)

Agent의 정체성, 역할, 그리고 답변의 톤앤매너를 정의합니다. 실제 LLM의 System Prompt에 들어갈 핵심 내용입니다.

| **항목** | **정의 내용** |
| --- | --- |
| **Agent 이름** | AI Wiki Assistant |
| **주요 역할** | 사용자의 자연어 질문을 이해하고, 승인된 AI Wiki를 검색(RAG)하여 시스템 기능, 업무 절차, 데이터 구조, 용어 등에 대해 정확하고 근거 기반의 답변을 제공하는 AI Assistant |
| **핵심 목표** | 사용자가 시스템 문서나 데이터 구조를 직접 찾지 않아도 필요한 정보를 빠르고 정확하게 이해할 수 있도록 지원하며, 승인된 Knowledge Base를 기반으로 신뢰성 높은 답변을 제공한다. |
| **톤앤매너** | 친절하고 전문적이며 간결하게 설명한다. 어려운 기술 용어나 업무 용어는 쉽게 풀어서 설명하고, 가능한 경우 관련 메뉴, 업무 절차, 출처를 함께 제공하여 사용자가 답변을 신뢰할 수 있도록 한다. |
| **제약 사항** | 승인되지 않은 Wiki는 사용하지 않는다.
검색 결과에 없는 내용을 추측하지 않는다. 
권한이 필요한 작업은 수행하지 않고 안내만 제공한다. 
정보가 부족한 경우에는 이를 명확히 알리고 추가 정보를 요청한다. |

### 워크플로우 및 오케스트레이션 (Workflow & Logic)

사용자 입력부터 최종 응답까지 Agent의 사고 과정과 행동 순서를 기술합니다.

**2.1 처리 로직**

- **Step 1 (Input Analysis):** 사용자의 자연어 질문을 분석하여 의도를 파악한다. 질문에 포함된 핵심 키워드, 메뉴명, 기능명, 설정명 등을 추출하여 검색 질의(Query)를 생성한다.
- **Step 2 (Tool Selection):** 분석된 의도에 따라 적절한 검색 도구를 선택한다.
- **Step 3 (Execution & Response):** 검색 결과를 통합하여 최신 기능 기준으로 답변을 생성한다.

**2.2 상태 관리**

- [대화 턴(Turn) 관리를 위한 상태 정의]
1. Question Received: 사용자 질문 수신
2. Intent Classified: 질문 유형 분류 완료

- 기능 문의

- 용어

- 변경 사항

1. Knowledge Retrieved: 관련 문서 검색 완료
2. Response Generated: 답변 생성 완료

포함 내용

- 질문에 대한 답변
- 관련 메뉴
- 업무 절차
- 참고 문서
- 최신 변경 사항
1. Escalation Required: 담당자 확인 필요
- 승인된 Wiki에서 검색 결과가 없는 경우
- 문서 내용이 부족한 경우
- 권한이 필요한 작업인 경우
1. Feature Request Required: 기능 요청 안내 필요

### 도구(Tools) 및 함수 명세 (Capability)

Agent가 외부 세상과 상호작용하기 위해 사용할 도구(Function Calling)를 정의합니다.

| **도구명 (Function Name)** | **기능 설명 (Description)** | **입력 파라미터 (Input Schema)** | **출력 데이터 (Output)** |
| --- | --- | --- | --- |
| analyze_question | 사용자의 자연어 질문을 분석하여 질문 의도(Intent)와 핵심 키워드를 추출한다. | question (String) | intent, keywords |
| retrieve_ai_wiki | AI Wiki(Vector DB)를 검색하여 질문과 관련된 문서를 조회한다. 내부적으로 Query Embedding, Vector Search, Metadata Filter, Top-K Retrieval을 수행한다. | keywords, intent | 관련 문서 목록(Top-K), 검색 점수 |
| rerank_documents | 검색된 문서를 관련도 및 최신성 기준으로 재정렬하여 최적의 문서를 선정한다. | retrievedDocuments | 재정렬된 문서 목록 |

### 지식 베이스 및 메모리 전략 (Context & Memory)

LLM이 기억해야 할 단기 대화 내역과 참조해야 할 장기 데이터(RAG) 전략을 수립합니다.

**4.1 RAG (검색 증강 생성) 전략**

- **참조 데이터 소스:** AI Wiki(기능 설명, 사용자 매뉴얼)
- ~~기능 메뉴얼, 운영 메뉴얼, 스토리보드 및 화면 설계서, 릴리즈 노트, 기능 변경 공지사항~~ [참조할 문서, DB, 매뉴얼 등]
- **청킹(Chunking) 방식:** 문서 구조 기반(Semantic + Hierarchical Chunking)을 적용한다.

- 제목(Header) 및 문단 단위로 분할

- 기능 단위 또는 메뉴 단위로 Chunk 생성

- 하나의 Chunk 크기는 **500~1,000 Token** 수준으로 구성

- Chunk 간 **10~20% Overlap**을 적용하여 문맥이 끊기지 않도록 구성

- 문서별 메타데이터(문서 유형, 버전, 작성일, 승인 여부)를 함께 저장하여 검색 정확도를 향상시킨다.

- **임베딩 모델:** text-embedding-3-large
- **Vector DB:** Qdrant

**4.2 대화 메모리 (Conversation History)**

- **메모리 유형:**

- 단기 메모리(Short-term Memory) : 현재 대화(Session) 내에서만 이전 질문과 답변을 유지하여 문맥을 이해한다.

| Session Memory | 대화 종료 시까지 유지 |
| --- | --- |
| Long-term Memory | 본 프로젝트 에서는 미적용 |
| 답변 근거 | Memory보다 Knowledge Base를 우선 사용 |
| 검색 우선순위 | 승인된 Wiki → Session Memory → LLM 응답 생성 |
- **저장 전략:**

- AI Wiki Builder는 다양한 시스템 문서를 AI가 이해하기 쉬운 Wiki 형태로 변환하여 Knowledge Base를 구축한다. 생성된 Wiki는 사용자 검수 및 승인 과정을 거친 후 AI Assistant의 답변 근거로 활용된다.

- 사용자별 **세션 단위**로 대화 이력을 관리[세션 유지 시간 및 초기화 기준]

| **구분** | **저장 방식** | **목적** |
| --- | --- | --- |
| 원본 문서 | Object Storage(MinIO/S3) | 원본 보관 |
| Wiki 문서 | PostgreSQL(Markdown) | 검수 및 버전 관리 |
| 임베딩(Vector) | Qdrant | 의미 기반 검색(RAG) |
| 메타데이터 | PostgreSQL | 문서 유형, 승인 상태, 태그, 생성일 등 관리 |

### 핵심 에이전트 기술 스택

일반적인 웹 개발 스택이 아닌, LLM의 답변 품질과 구조를 제어하기 위한 기술적 의사결정을 기입합니다.

| **구분** | **선정 전략/기술** | **선정 사유 (논리적 근거)** |
| --- | --- | --- |
| **LLM Model** | Azure OpenAI GPT-4.1 | 사용자 질문 이해 및 자연어 응답 생성 |
| **Agent Framework** | LangGraph | Wiki 생성 및 질의응답 Workflow 구성 |
| **Prompt Strategy** | ReAct + RAG | 검색된 AI Wiki 문서를 근거(Context)로 활용하여 답변을 생성 |
| **Output Parsing** | Structured Output (JSON) | 답변, 참고 문서, 출처, 신뢰도 등을 정형화된 구조로 반환하여 UI 표시 및 후속 시스템 연계가 용이함 |
| **Monitoring** | LangSmith | Prompt 실행 과정, Tool 호출, 토큰 사용량, 응답 시간 및 Workflow를 추적하여 디버깅과 성능 분석을 지원 |
